# Лабораторна робота №3: Реалізація власного blockchain (DC_Lab_3)

## Огляд та призначення продукту
Цей проєкт є спрощеною реалізацією платіжної системи на базі технології блокчейн. Це навчальна модель, створена для вивчення фундаментальних принципів розподілених реєстрів.

**Цілі проєкту:**
* Розуміння механізмів зв’язування блоків через хеш-функції.
* Реалізація власної функції хешування (SHA-1) «з нуля».
* Забезпечення цілісності даних: зміна одного блоку робить недійсною всю наступну чергу.
* Вивчення базової логіки транзакцій та управління балансами користувачів.

## Склад системи
Система працює з однією основною валютою — **TC (TestCoin)**.
* **CoinDatabase:** глобальне сховище поточних балансів усіх користувачів.
* **BlockHistory:** незмінний список блоків, що утворює ланцюжок.
* **TxDatabase:** реєстр ідентифікаторів транзакцій для захисту від повторного використання.

## Взаємодія
Продукт реалізований як консольний додаток мовою **C#**. 
* Весь ввід та вивід здійснюється через термінал.
* Логіка системи розділена на незалежні класи (`Account`, `Block`, `Transaction`, `Operation`, `Hash`), що забезпечує чистоту архітектури.

## Функції продукту
1.  **Управління акаунтами:** Створення користувачів з унікальними ідентифікаторами (GUID) та іменами.
2.  **Faucet (Кран):** Механізм нарахування початкових тестових монет користувачам.
3.  **Проведення платежів:** Переказ TC між акаунтами з автоматичною перевіркою достатності коштів.
4.  **Кастомне хешування:** Використання власної реалізації алгоритму SHA-1 для генерації ID блоків та транзакцій.
5.  **Формування ланцюжка:** Групування транзакцій у блоки, де кожен блок містить `PrevHash` (хеш попереднього блоку).
6.  **Аудит системи (PrintBlockchain):** Візуалізація всього ланцюжка блоків у консолі для перевірки зв'язності та цілісності.

## Вимоги до безпеки
* **Зв'язність блоків:** Кожен блок жорстко прив'язаний к хешу попереднього. Будь-яка спроба змінити дані в минулому зруйнує ланцюжок.
* **Валідація транзакцій:** Операція відхиляється, якщо у відправника недостатньо коштів.
* **Захист від дублювання:** Використання поля `nonce` у транзакціях гарантує унікальність хешу навіть при однакових сумах переказу.

## Характеристики користувачів
У демонстраційному сценарії беруть участь три користувачі:
* **Andrii** — основний відправник, що отримує монети з крана.
* **Roma** — посередник, що бере участь у ланцюжку переказів.
* **Oleg** — кінцевий отримувач активів.

## Обмеження
* **Локальність:** Система працює в оперативній пам'яті (без децентралізації та мережевої взаємодії).
* **Відсутність майнінгу:** Консенсус досягається шляхом прямої валідації при додаванні блоку.
* **Криптографічні підписи:** У цій версії не використовуються (ідентифікація відбувається за ID акаунту).
* **Зберігання даних:** Стан блокчейну скидається після завершення роботи програми.
